Onspace — Vision & Master Plan

Multi-product SaaS for SME — CRM, ERP, Accounting, Restaurant, Super Shop — single admin panel, offline-first, BD-friendly cost.

build: solo + Claude Opus 4.7 deal: offline stage: pre-MVP

1 · Original briefs — refined

User prompts combined and cleaned. Editable.

Brief A — Hosting for dev/demo

Need free or cheap hosting for OnspaceCRM (Next.js + NestJS + Postgres + Redis + Python Playwright) so client can review production-like environment. Hostinger subdomain available for frontend dist. Need free dev solution for backend and DB. Same stack will run wider SaaS portfolio (CRM, ERP, restaurant, accounting) under one shelter — subscription-based service.

Brief B — Multi-tenant SaaS shape

Sell web apps to companies: restaurants, super shops, services. All products live under one master admin panel. From the admin panel we create new customer workspaces, decide how many users they can invite, and select which features they can access. Multi-user + role-based access inside each workspace. Easy to plug new product/feature later.

Brief C — Performance & payment

System must feel lightspeed for company use — no click lag, no jitter, no bugs. No online payment for now: subscription deals close offline; we manually configure workspace + features after agreement; customer logs in and works online.

Brief D — Build approach & infra constraints

Build all products first, then sell manually to targeted audience (SME, eventually large enterprise reference letters). Entire build driven by Claude Opus 4.7 + manual polish. Must scale, but must run on a low-cost Bangladeshi server budget. Web app must keep working offline; data syncs on reconnect.

2 · Vision & North Star

One sentence we anchor every decision to.
Onspace is a single, locally affordable SaaS platform that gives every SME in Bangladesh — and later the region — production-grade business software (CRM, ERP, Accounting, Restaurant, Super Shop) under one login, controlled centrally by us, that keeps working even when their internet does not.

Three-year ambition

  • 10,000+ SME workspaces on the platform.
  • ≥ 5 in-house products, all sharing one core (auth, billing, RBAC, audit, sync).
  • Offline-first PWA usable from a phone in a market stall.
  • Servers, support, language and pricing tuned for Bangladesh first, then SAARC.

Frontend shape

  • Single Next.js app — shell renders only the products+features the workspace has.
  • Per-product code-split bundle so cold load is small.
  • Admin panel = route group /admin guarded by user.isAdmin.

6 · Admin Panel — what it controls

  • Workspaces: create, rename, suspend, delete, impersonate.
  • Products per workspace: tick which products are enabled (CRM, ERP, …).
  • Features per workspace: inside each product, tick allowed features (e.g. CRM email send on/off).
  • User seats: set max user count per workspace.
  • Roles: default roles seeded; admin can craft custom roles for a workspace.
  • Subscription: plan name, start/expiry date, status, notes (deal terms).
  • Branding: per-workspace logo, color, business name on invoices.
  • Audit log: every write — who / what / when, filterable.
  • Usage: DB rows used, storage used, API calls — for capacity planning.
  • Health: failed jobs, slow queries, sync backlog per workspace.

7 · Tenancy & data isolation

Chosen model: shared Postgres database + workspaceId column on every tenant row + Postgres Row-Level Security policies.

  • One DB cluster → one cost, one backup, one migration.
  • Every Prisma query auto-scoped via middleware (SET LOCAL app.workspace_id).
  • RLS enforces isolation even if a developer forgets a where.
  • Scales to thousands of SMEs on a single cheap VPS.
  • Optional escape hatch later: heavy/enterprise tenant → dedicated DB by flipping a tenant.dbUrl field.

8 · RBAC & feature gating

Permission strings shaped as <product>.<resource>.<action> — e.g. crm.lead.write, erp.invoice.approve.

Three-layer gate on every request

  1. Subscription guard: workspace subscription active and not expired.
  2. Feature guard: workspace has this feature enabled.
  3. Permission guard: user's role includes this permission string.

Default roles seeded per workspace

  • Owner — all permissions, billing visibility.
  • Admin — all except billing.
  • Manager — per-product manage rights.
  • Staff — read + create, no destructive ops.
  • Viewer — read-only.

Custom roles editable from Admin Panel + from workspace owner panel (limited).

9 · Offline-first & sync

Critical for Bangladeshi SMEs — power cuts, weak mobile data, market-stall use. Plan:

Client side

  • Next.js installable PWA + service worker (Workbox).
  • Local store: IndexedDB via Dexie mirroring tables the user touches (leads, invoices, products, customers).
  • Every mutation written first to local DB, then queued to outbox table with UUIDv7 + clientTimestamp.
  • UI reads from local store → zero-latency clicks even online.

Sync engine

  • On reconnect, outbox flushed to POST /sync/push in batches.
  • Server returns canonical IDs + version vector.
  • Server pushes server-side changes via GET /sync/pull?since=cursor (long-poll or SSE).
  • Conflicts → last-write-wins by default, manual resolve UI for high-value entities (invoices).
  • Each row has updatedAt + version + updatedBy; deletes are soft (tombstones) so clients can converge.

10 · Performance — lightspeed feel

  • Index every workspaceId, every FK, every sortable column.
  • PgBouncer / Neon pooler in front of Postgres.
  • Redis cache for hot reads — workspace settings, feature flags, role permissions (TTL 60s).
  • API budget < 200 ms p95.
  • Frontend: React Server Components for first paint; TanStack Query for client cache + optimistic updates.
  • Virtualized lists for > 100 rows (@tanstack/react-virtual).
  • Code-split per product bundle.
  • BullMQ + Redis for anything > 50 ms (email send, exports, scrape, reports).
  • Pino slow-query log + Sentry to catch lag before customer does.
  • Gzip/Brotli, HTTP/2, CDN in front (Cloudflare free) for static.

13 · Subscription — fully offline

No Stripe, no online gateway. Sales close offline (in-person / phone / bKash). Admin enters the deal:

  • Subscription table — planName, startsAt, expiresAt, status, amountPaid, notes.
  • Daily cron flips expired subscriptions to suspended; blocks API and shows renewal banner.
  • Receipt PDF auto-generated and downloadable from admin panel.
  • Renewal reminders — system notifies us 14 / 7 / 1 days before expiry.

Later we can flip on bKash / SSLCommerz when ready — same table, just add gatewayTxId.

14 · Security & compliance

  • Passwords: bcrypt cost 12 minimum.
  • JWT short access token (15 min) + refresh token rotation; revoke on logout.
  • Rate-limit auth + sensitive endpoints (nestjs-throttler + Redis).
  • Postgres RLS on every tenant table — defense in depth.
  • All writes go through audit log.
  • HTTPS only (Caddy auto-TLS via Let's Encrypt).
  • Daily encrypted pg_dump → off-site (R2).
  • Secrets in .env on server, never in repo; use SOPS later.
  • Impersonation only via admin panel, always audit-logged + banner shown to impersonated user.
  • Per-tenant data export endpoint (GDPR-style, useful for trust).

15 · OnspaceCRM → product module changes

OnspaceCRM today is standalone. To slot under the Admin Panel as the crm product:

  1. Add platform-core Prisma models (User, Workspace, WorkspaceMember, Role, Product, Feature, WorkspaceProduct, WorkspaceFeature, Subscription, AuditLog).
  2. Add workspaceId String to every existing CRM table; backfill current data into a single "internal" workspace.
  3. Add composite indexes (workspaceId, createdAt) + workspaceId on every FK.
  4. Add Auth module — register / login / refresh / logout, bcrypt + JWT.
  5. Add Tenant guard middleware — extracts workspaceId from JWT, sets Postgres session var, attaches to req.
  6. Add Prisma client extension auto-injecting workspaceId on all queries.
  7. Migration: enable RLS + tenant-isolation policy on every CRM table.
  8. Add Feature-flag service + @RequireFeature('crm.email') + @RequirePermission('crm.lead.write') decorators.
  9. Register CRM as a Product seed + seed feature keys (crm.lead, crm.group, crm.email, crm.scraper, crm.task).
  10. Wrap each CRM endpoint with the three-layer guard.
  11. Frontend: read enabled products + features from /me response → render sidebar conditionally.
  12. Hide CRM nav entirely if workspace has no crm product.
  13. Add per-workspace settings page (branding, default currency, timezone).
  14. Wire offline outbox + Dexie store for Lead, Note, Task tables.
